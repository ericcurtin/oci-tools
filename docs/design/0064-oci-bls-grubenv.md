# Design note 0064: `oci-bls` grubenv read/write (milestones 5/6)

Status: implemented (the low-level grubenv block itself; BLS entry
files and the higher-level boot-counting/rollback logic built on top
of this are separate, later follow-ups — see "What's still not here")
Scope: `crates/oci-bls/src/grubenv.rs` (new), `crates/oci-bls/
src/lib.rs`, `crates/oci-bls/Cargo.toml`, `ci/vm-prepare.sh`;
`crates/oci-erofs/src/verity.rs` (an unrelated, small test-reliability
fix found and fixed along the way — see "A real, rare flake found and
fixed while verifying").

`oci-bls` was, until now, a 17-line stub with nothing but a planning
doc comment — this is its first real capability: the GRUB environment
block (`grubenv`) grub2 itself uses for `saved_entry` (which boot
entry is the default) and boot-counting variables (`boot_success`,
`boot_indeterminate_count`). `ociboot`'s own planned boot-
counting/rollback logic (milestone 6) will read and write the exact
same file.

## No written spec exists; every detail verified against the real `grub-editenv` binary directly

Unlike BLS entries (which the uapi-group does publish a real written
spec for), grubenv has no such document — grub2's own C implementation
is the only real authority. Every detail in this increment was
confirmed by direct, hands-on comparison against the real
`grub-editenv` binary (`grub2-common`, already installed on this
development host), not guessed from memory or inferred from a header
comment:

* The exact, literal 94-byte header:
  `"# GRUB Environment Block\n# WARNING: Do not edit this file by
  tools other than grub-editenv!!!\n"`.
* `grub-editenv create`'s own default total size is exactly 1024
  bytes, but the real tool honors whatever size an *existing* file
  already is when updating it — confirmed by hand-shrinking a real
  file to 194 bytes and updating it with the real tool, which left it
  exactly as short as before rather than re-padding back up to 1024.
* Updating an *existing* key preserves its original line position;
  a brand new key is appended after the last existing entry; `unset`
  removes the line and shifts everything after it up — all confirmed
  directly via the real tool's own `set`/`unset`/`list` output.
* The real tool's own `"error: environment block too small"` — it
  genuinely refuses to write if the header plus every `key=value\n`
  line wouldn't fit in the block's current total size, confirmed by
  reproducing the exact error with an oversized value.
* Values may contain `=` themselves (e.g. `options=root=UUID=... rw`)
  — confirmed the real tool splits only on the *first* `=`.

## A real tool oddity deliberately *not* replicated, after extensive direct testing

The obvious temptation was to replicate the real tool's own tolerance
for malformed content after the header byte-for-byte. Extensive direct
testing (a dozen or so hand-crafted files with various combinations of
stray non-`key=value` lines, embedded `#` lines, and trailing garbage,
each compared against the real tool's own `list` output) never reduced
to one simple, obviously-intentional rule — some non-matching lines
get echoed back if a later line still contains `=`, others don't,
depending on details that don't matter for this crate's real job.
**A genuine grubenv file — whether written by the real `grub-editenv`
or by this module — never contains such content in the first place**:
the header is always immediately followed by zero or more well-formed
`key=value\n` lines and then, immediately, the `#` padding run, with no
other shape ever possible. [`grubenv::parse`] matches that real,
well-formed shape exactly (confirmed via direct round-trip against
files the real tool itself produced) and simply stops at the first
line that doesn't parse as `key=value` — exactly the boundary a
well-formed file's own padding run creates. Chasing byte-for-byte
parity with the real tool's own undocumented tolerance for adversarial
input a real grubenv file never actually contains would have been
needless complexity and needless risk of a subtle bug, not a real
compatibility requirement.

## One deliberate improvement over the real tool: atomic writes

`strace`-confirmed directly: real `grub-editenv` opens the target path
with `O_WRONLY|O_CREAT|O_TRUNC` and writes straight into it — no
temporary file, no rename, no atomicity at all. A crash or power loss
mid-write can leave a real, booted machine's own grubenv truncated or
corrupted, which is a genuinely bad place to be for a file that gates
which OS deployment boots next. [`grubenv::write`] instead writes to a
`tempfile::NamedTempFile` in the same directory and renames it into
place, so a caller only ever observes the fully-old or fully-new
content, never a torn write. This is exactly the kind of "beat the
equivalent" improvement this project's own stated goals call for,
applied somewhere it has a real, meaningful safety payoff rather than
purely for its own sake.

## Real, automated tests

10 tests in `oci-bls::grubenv`: four compare directly against the real
tool (a block this module writes is accepted and correctly read back
by the real `grub-editenv`'s own `list`; a file the real tool itself
created round-trips through this module's own `read`/`to_bytes` byte
for byte; updating an existing key via this module preserves its
position exactly like the real tool; the real tool's own default
1024-byte size and exact header bytes are checked directly) — each
tries both `grub-editenv` (Debian/Ubuntu's own binary name) and
`grub2-editenv` (Fedora/CentOS Stream's own binary name, confirmed to
be genuinely different between the two distro families, no shared
alias), skipping itself with a clear message if neither is installed.
The rest are pure logic, no real tool needed: `unset` plus a
subsequently-appended key; refusing to overflow the block's own size;
rejecting a missing/wrong header; a header-only block with zero
entries; atomicity (a failed serialization never touches the real
file on disk); a value containing its own `=` round-trips intact.

## CI: `grub2-tools` (CentOS Stream 10) / `grub-common` (Ubuntu 26.04) added to both VM base images

Neither VM base had a grubenv tool installed before this. Added to
`ci/vm-prepare.sh` alongside 0061's `erofs-utils` and 0063's
`cryptsetup`/`cryptsetup-bin`, so these tests run for real inside CI.

## A real, rare flake found and fixed while verifying (unrelated to this increment's own scope)

While re-running the full workspace test suite repeatedly to confirm
this change hadn't broken anything (as required every turn), a real,
if rare, failure surfaced in `oci-erofs::verity`'s own
`sealed_file_becomes_immutable` test — nothing to do with `oci-bls`,
which doesn't touch `oci-erofs` at all. The real fs-verity
`FS_IOC_ENABLE_VERITY` ioctl genuinely refuses to enable while *any*
file descriptor anywhere still has the target file open for writing
(so its own content can't change out from under the Merkle tree being
built), returning `ETXTBSY` ("Text file busy") — and on this shared,
multi-tenant development host, a freshly written, already-closed test
file's own `enable` call hit exactly that error twice across many
otherwise-clean runs, almost certainly some other process on the host
(e.g. a filesystem indexer) briefly opening it. [`crate::verity::enable`]
itself was correct either way (`ETXTBSY` is a real, well-documented
kernel condition, not a bug) — but the *test* wasn't robust to it. A
handful of retries with a short backoff (`enable_retrying_on_text_file
_busy`, test-only, not part of the crate's own public API) is exactly
how a real caller like `ociboot` should also handle this documented,
transient condition — confirmed to fix the flake via 10 consecutive
clean stress-test runs afterward, where it had failed twice across the
prior handful of runs.

## Performance

This increment touches only the still-mostly-stub `oci-bls` crate,
`ci/vm-prepare.sh`, and a test-only retry helper in `oci-erofs::verity`
— `oci-runtime-core`/`ocirun`/`ociman run`'s own hot paths are
untouched (confirmed via `git diff --stat`), and neither `oci-bls` nor
`oci-erofs` is linked into any binary's own hot path yet, so no
benchmark re-verification was needed.

## What's still not here

* BLS entry files themselves (`title`/`version`/`linux`/`initrd`/
  `options` under `/boot/loader/entries/`) — a real written spec
  exists for these (unlike grubenv), a separate, cleanly separable
  next increment.
* The higher-level boot-counting/rollback/atomic-default-entry-flip
  logic `ociboot`'s own milestone 6 needs, built on top of this
  primitive rather than duplicating it.
* Kernel argument (kargs) editing.
* `grub2-mkconfig`/`grub-install` wrapped behind a trait (this
  increment's own scope was entirely the grubenv file format itself,
  which needs neither).
